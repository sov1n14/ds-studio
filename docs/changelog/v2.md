# v2.x 版本變更記錄

## [2.9.0] - 2026-06-07

### Added

- **Censor Reply Restore now monitors the `edit_message` API** (`content/censor-xhr-hook.js`): Editing a previously-sent user message regenerates the reply through POST `/api/v0/chat/edit_message` instead of `/api/v0/chat/completion`, but the XHR hook's URL filter only matched the latter — censored replies produced by an edit were never intercepted, so the DOM element was detected as censored yet no fragment ever arrived and the content stayed unrestored (field-confirmed: the edited reply logged "Detected censored message" with no corresponding "Fragment complete!" and no storage match). Field captures verified that `edit_message` is SSE with a format identical to `completion` (same `event: ready` carrying `response_message_id`, same `data:` patch syntax, same `CONTENT_FILTER` status paths) and that its request body uses the same `chat_session_id` and `prompt` field names (its extra `message_id` field is the edited USER message's id and is deliberately ignored — the reply id still comes from the SSE stream). The hook now matches URLs against a new `INTERCEPTED_ENDPOINTS` list (`/api/v0/chat/completion`, `/api/v0/chat/edit_message`) via the pure helper `getMatchedEndpoint(url)`, and the interception log states which endpoint was matched. Request-body parsing, `sse-parser.js`, `censor-reply-restore.js`, and the `DVOY_FRAGMENT_COMPLETE` payload are all unchanged — the downstream pipeline is API-agnostic.

### Changed

- **Unit tests** (`test/unit/censor-xhr-hook-edit-message.spec.js`, new file): 24 new tests covering endpoint matching (both endpoints accepted; `/api/v0/chat/history`, `/api/v0/file/upload`, and empty URLs pass through un-hooked), endpoint-labelled interception logging, end-to-end edit_message SSE parsing modeled on the field capture (messageId taken from the stream's `response.message_id` — not the request's `message_id` —, `censored`/`finished` flags, TEMPLATE_RESPONSE exclusion, THINK accumulation, `thinking_elapsed_secs`), request-body extraction for both endpoints (including missing/non-string/non-JSON bodies), and dual-endpoint coexistence (one completion + one edit_message → two `DVOY_FRAGMENT_COMPLETE` dispatches). Suite: 829 tests total, all passing.

## [2.8.12] - 2026-06-07

### Fixed

- **Censor Reply Restore injecting wrong content within the same chat (stale pending-queue entry)** (`content/censor-reply-restore.js`): `_pendingQueue` entries were only ever removed by the queue-fallback `.shift()` in `_getMessageIdFromElement()`. When an element resolved via the map or storage path instead, the matching queue entry silently survived. Field-observed failure (confirmed via console/DOM evidence): a censored element appeared before its fragment completed and stayed unrestored; the fragment then arrived (queue `[2]`, record saved) and the re-scan resolved the element via the storage path — leaving the stale `2` in the queue; when the chat's next censored element later appeared before its own fragment, map and storage both missed (its record didn't exist yet) and the queue fallback blindly popped the stale `2`, injecting the first reply's content into the third reply's element. Once its real fragment arrived the element was already marked restored, so the wrong content persisted until a refresh reset memory state. Two-part fix: (1) resolving a messageId via the map or storage path now purges that id from `_pendingQueue` (all occurrences, logged); (2) the queue fallback validates before consuming — if the element yields a prompt key via `_getPrecedingUserPromptKey()` and the queue-head candidate's stored record carries a mismatching `prompt_key`, the fallback rejects (logged), leaves the queue untouched, and returns null so the element is restored correctly by the re-scan after its own fragment arrives; blind consumption is preserved only when no prompt key is obtainable or the candidate has no stored record yet.

### Changed

- **Unit tests** (`test/unit/censor-reply-restore.spec.js`): 9 new tests covering map-path and storage-path queue purging (including no-op when the id is absent), queue-fallback rejection on prompt mismatch (queue must remain unconsumed), the three fallback-acceptance paths (prompt match, no stored record yet, element without an obtainable prompt key), and the full stale-queue field scenario end-to-end. Suite: 805 tests total, all passing.

## [2.8.11] - 2026-06-07

### Fixed

- **Censor Reply Restore injecting wrong content during live conversations (cross-chat contamination)** (`content/censor-reply-restore.js`): DeepSeek's web app is an SPA — switching chats never reloads the page — yet three pieces of CensorRestore runtime state were neither chat-scoped nor cleared on navigation. (1) `_restoredMessages` was keyed by bare `message_id`, but message ids are per-chat sequence numbers (2, 4, 6, …), so a new chat's censored record silently overwrote the previous chat's record under the same key, and injection lookups never validated the session. (2) `_keyToMessageId` and `_pendingQueue` survived chat switches while `data-virtual-list-item-key` values are per-chat indices reused across chats, so a stale map entry from the previous chat hijacked the new chat's element at the same virtual index. Observed result (confirmed via console/DOM evidence): the first censored message of a new chat displayed the previous chat's restored content, and a later censored message displayed the current chat's first response — self-healing only after a refresh because memory state reset. The fix: all `_restoredMessages` reads/writes now use session-scoped keys via the new `_recordKey(sessionId, messageId)` helper (`"<session>::<id>"`, falsy session → `nosession`); legacy bare-key records are migrated once on load using their embedded `chat_session_id`; a record with a falsy `chat_session_id` (or a current URL without a session id) can never match an element; and the new `_checkSessionChange()` — invoked at the entry of `_tryRestoreMessage()`, `_onFragmentComplete()`, and `_tryRestoreFromStoredRecords()` — clears `_keyToMessageId` + `_pendingQueue` and resets `_storedRecordsApplied` whenever the URL session changes (null → non-null transitions preserve the queue, since a brand-new chat's first fragment may arrive before the URL settles).

### Changed

- **Diagnostic logging** (`content/censor-reply-restore.js`): `_getMessageIdFromElement()` now logs which path resolved each message id (`map` | `storage` | `queue`) — the previous single `Found messageId:` log could not distinguish them, which was a blind spot during this investigation; `_checkSessionChange()` logs every session transition that clears runtime state.
- **Unit tests** (`test/unit/censor-reply-restore.spec.js`): 23 new tests covering `_recordKey()` formatting, cross-chat record coexistence (two chats both holding `message_id=2`), legacy-key migration (including null-session records becoming permanently unmatchable `nosession::` entries), null-session match strictness, all four `_checkSessionChange()` clearing rules, the SPA contamination regression scenario (stale virtual-list key must not inject the previous chat's content), and the live-XHR + refresh-restore happy paths under session-scoped keys; 6 existing tests updated from bare-key to session-scoped assertions. Suite: 796 tests total, all passing.

## [2.8.10] - 2026-06-07

### Fixed

- **Censor Reply Restore not re-applied after page refresh** (`content/censor-reply-restore.js`): Stored records loaded correctly at startup, but the only code path that matched stored records against the DOM — `_tryRestoreFromStoredRecords()`, invoked exactly once from `enable()` → `applyToExisting()` — ran before React rendered the chat history (the startup window is occupied by the page's own WASM/React initialization), found zero `.ds-message._63c77b1` elements, silently returned, and was never retried. When the MutationObserver later detected the censored messages, `_getMessageIdFromElement()` consulted only the runtime `_keyToMessageId` map (empty after a refresh) and `_pendingQueue` (populated by live XHR only), so old messages logged "No messageId available (pendingQueue empty, no storage match)" and stayed unrestored. The fix adds a storage fallback to the observer path: new `_resolveMessageIdFromStorage()` matches a single element against stored records by URL session id + normalized preceding-prompt key (writing into `_keyToMessageId` on a hit, never touching the queue); `_getMessageIdFromElement()` lookup order is now `_keyToMessageId` → storage → `_pendingQueue.shift()` as a true last resort — which also fixes a mis-assignment risk where a freshly generated message's queued id could be claimed by an older unrestored element; and when all per-element lookups miss, `_tryRestoreMessage()` triggers a full `_tryRestoreFromStoredRecords()` scan, guarded by the new `_storedRecordsApplied` flag (reset on every `_onFragmentComplete`) so repeated observer mutations don't rescan needlessly. The live-XHR happy path is unchanged.

### Changed

- **Diagnostic logging** (`content/censor-reply-restore.js`): `_tryRestoreFromStoredRecords()` — previously completely silent — now logs the extracted session id, unrestored-censored element count, per-element prompt keys, stored vs session-matched record counts, per-prompt-key match outcomes, and each injection, all under the `[DV:CensorRestore]` prefix.
- **Unit tests** (`test/unit/censor-reply-restore.spec.js`, `test/unit/censor-xhr-hook.spec.js`): 16 new tests covering `_getMessageIdFromElement()` lookup order (storage precedence over a non-empty queue, queue-not-consumed guarantees), `_resolveMessageIdFromStorage()` (session/prompt matching, already-claimed id skipping, queue immutability), the post-refresh observer-triggered restore scenario, double-injection idempotency, `applyToExisting()` cold start, `_storedRecordsApplied` guard semantics, and `_tryRestoreFromStoredRecords()` edge returns; the two `censor-xhr-hook` tests that read deleted YAML fixtures under `to-fix/` were rewritten with inline SSE strings. Suite: 773 tests total, all passing.

## [2.8.9] - 2026-06-07

The `_getToolbarGroup()` diagnostic count and `.ds-flex` fallback loop use the same combined selector (primary `.ds-flex._965abe9` selector unchanged — still valid in the new DOM).

### Fixed

- **GoToTop button broken by DeepSeek's `ds-button` redesign** (`content/go-top.js`, `content/go-top.css`): DeepSeek replaced the native go-bottom button — formerly `<button class="ds-atom-button ds-floating-button …">` — with a new design-system element: `<div role="button" class="ds-button ds-button--outlinedNeutral ds-button--outlined ds-button--circle ds-button--m ds-button--icon-relative-m ds-button--floating _0706cde">` carrying inline CSS variables (`--dsl-button-height: 34px`, `--dsl-button-icon-size: 14px`, floating fill/hover colors) and three child layers (`.ds-button__background`, `.ds-button__border`, `.ds-button__icon`). ### Changed

## [2.8.6] - 2026-06-06

### Fixed

- **GoToTop button missing the native white border** (`content/go-top.css`): The base `.dsw-gotop` rule declared `border: none`, `background`, `color`, and flex-centering properties that shadowed the site's `ds-*` component styling (extension CSS loads after the page stylesheet at equal specificity), suppressing the native button's subtle border (`var(--dsw-alias-border-inverted)`). The base rule now keeps only behavior-neutral declarations (`cursor`, `pointer-events`, `transition`) so the copied `ds-*` classes fully drive the appearance; the `position: fixed` last-resort mode gained its own self-sufficient appearance (background, color, border, shadow, flex centering) since it carries no `ds-*` styling context.
- **GoToTop button flickering after click** (`content/go-top.js`): Scrolling up after a click mounts the native go-bottom button, which triggered a solo → stacked mode transition that removed the button element and re-created it as `display: none` — a visible flicker, repeated as the native button mounted/unmounted mid-scroll. New `_transitionToStacked()` / `_transitionToSolo()` helpers now REUSE the same element (moved via `insertBefore`, modifier classes swapped, inline offsets recomputed or cleared, `display` state preserved), with a no-op guard when the mode is already correct; genuine re-injection (element removed from DOM by React) restores the prior visibility state immediately.
- **GoToTop scrolling only ~one message group instead of to the very top** (`content/go-top.js`): `_isAtTop()` trusted loose "first mounted message" anchor selectors; with DeepSeek's virtual list the true first message is usually unmounted, so the first *mounted* message near the viewport top yielded a false "at top" and the scroll loop exited after roughly one viewport. `_isAtTop()` now requires verifiable evidence only: `scrollTop <= 1`, or the verifiable first-message anchor `[data-virtual-list-item-key="1"]` fully within the viewport.

### Added

- **Unit test coverage** (`test/unit/go-top.spec.js`): New describe blocks for `_transitionToStacked` / `_transitionToSolo` (element reuse, class swap, DOM position, offset handling, display preservation), observer no-op guards, and a rewritten `_isAtTop` matrix (scrollTop 0/1/2 boundaries, verifiable vs untrusted anchors). Also fixed a pre-existing flaky fake-timer test. Suite: 148 tests in `go-top.spec.js`, 731 tests total, all passing.

## [2.8.5] - 2026-06-06

### Fixed

- **GoToTop pinned to the page corner on initial load** (`content/go-top.js`, `content/go-top.css`): On initial page load the native go-bottom button is not rendered, so the previous two-path strategy degraded straight to the `position: fixed` bottom-right fallback. A new "solo mode" middle path was added: when the native button is absent but its wrapper container (`aaff8b8f`, located by the new `_locateWrapperDirect()` with structural fallback `._871cbca > div:nth-child(2)`) exists, the button is injected directly into the wrapper (`_injectIntoWrapperDirect()`, `_injectionMode = 'wrapper-solo'`) and positioned at the native button's own spot. The wrapper observer now performs bidirectional solo ↔ injected transitions as the native button mounts/unmounts.
- **GoToTop solo-mode button shape** (`content/go-top.js`, `content/go-top.css`): When the native go-bottom button was absent, the injected go-top button rendered as a vertical oval instead of a circle identical to the native button. `_injectIntoWrapperDirect()` previously overwrote the button's className, discarding the fallback `ds-*` classes that carry the site's circular button recipe; it now appends the `dsw-gotop--solo` modifier via `classList.add`, and `_createButtonElement()` applies the native button's inline sizing (`padding: 9px; font-size: 14px; line-height: 22px`) to every created button. The `.dsw-gotop--solo` CSS rule is reduced to positioning only (`position: absolute; bottom: 100%; right: 12px; margin-bottom: 20px`).
- **GoToTop overlapping the native go-bottom button** (`content/go-top.js`, `content/go-top.css`): When the native button appeared, the go-top button overlapped it because both resolved to the same absolute position and the old `margin-bottom: 8px` sibling-flow spacing had no effect on absolutely positioned elements. Injected mode now adds a `dsw-gotop--stacked` class (`position: absolute; bottom: 100%; right: 12px`) and the new `_applyStackedOffset()` helper computes the inline `margin-bottom` from the native button's runtime geometry (native margin-bottom + native height + 8px gap; CSS fallback 68px), so the go-top button always sits 8px above the go-bottom button. Mode transitions (solo ↔ stacked) swap the modifier classes and recompute or clear the inline offset.

### Added

- **Unit test coverage** (`test/unit/go-top.spec.js`): New tests for `_applyStackedOffset` (jsdom fallback and mocked-geometry paths), native-matching inline styles on created buttons, `ds-*` class preservation in solo mode, the `dsw-gotop--stacked` class and computed margin in injected mode, and bidirectional solo ↔ stacked mode transitions. Suite: 123 tests in `go-top.spec.js`, 706 tests total, all passing.

## [2.6.0] - 2026-05-29

### Added

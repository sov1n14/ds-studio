## [3.2.1] - 2026-06-13

### Added

- **Edit-message cleanup** (`content/edit-message-cleanup.js`, new): When the user clicks DeepSeek's edit button on a previously-sent message, the edit textarea is pre-filled with the FULL injected payload (`<system-prompt>...</system-prompt>` + `<user-input>\n...\n</user-input>` wrapper). This module strips the wrapper so the user edits only their original text. A single document-level delegated click listener resolves the edit button via `e.target.closest('.d4910adc')` (the obfuscated edit-button class). On an edit click it (1) removes the `max-height` constraint from every `.cc852ac5` and `._646a522` element by setting inline `style.maxHeight = 'none'` (not restored afterward), and (2) locates the edit textarea — which renders asynchronously — by walking up from the button to the enclosing message container (`findMessageContainer`) then watching via `MutationObserver` with a 2000ms hard timeout (`DETECTION_TIMEOUT_MS`; synchronous path used when the textarea already exists). Once found, if the value matches `/<user-input>\n([\s\S]*)\n<\/user-input>$/` the textarea is rewritten with ONLY the inner content via the native value setter + dispatched `input`/`change` events (same React-aware technique as `injectPrefix`/`quote-reply.js`). If NO `<user-input>` wrapper is present, the textarea content is left completely untouched (explicit requirement). Listener registration is idempotent via a window guard; per-click observer/timeout state is kept local to the handler closure (no module-level mutable state). Exports for testing: `extractUserInput`, `removeMaxHeightConstraints`, `applyTextareaCleanup`, `findMessageContainer`, `waitForTextareaInContainer`, `handleEditButtonClick`, plus constants `EDIT_BUTTON_CLASS`, `MAX_HEIGHT_SELECTORS`, `USER_INPUT_REGEX`, `DETECTION_TIMEOUT_MS`.
- **Unit tests** (`test/unit/edit-message-cleanup.spec.js`, new): 50 tests across 6 groups — `extractUserInput` (real injected shape, multi-line, end-anchor strictness, null/non-string guards), `removeMaxHeightConstraints` (both selectors, unrelated elements untouched, root scoping), `applyTextareaCleanup` (rewrite + `input`/`change` dispatch on match, critical no-op + no event on plain text), `findMessageContainer` (closest ancestor with textarea, null cases), `waitForTextareaInContainer` (sync path, async observer path, fires-once, timeout auto-disconnect via fake timers), and `handleEditButtonClick` (non-matching no-op, max-height removal, wrapped cleaned, plain untouched). Full suite: 957 tests across 49 files, all passing.
- **Manifest**: Registered `content/edit-message-cleanup.js` in content_scripts (after `content-script.js`).

- **Manifest**: Version 3.2.0 → 3.2.1.

## [3.2.0] - 2026-06-09

### Added

- **Mobile sidebar swipe gesture** (`content/mobile-sidebar-swipe.js`, new 372 lines): Detects right-swipe gestures within the central 80% viewport area on mobile devices and clicks the sidebar toggle button to show/hide the navigation sidebar. Gesture recognition uses five conditions: deltaX >= 50px (SWIPE_THRESHOLD_PX), horizontal dominance (deltaX > |deltaY| * 1.5), duration < 500ms (SWIPE_MAX_DURATION_MS), startX within center 80% horizontal zone, startY within center 80% vertical zone — where the trigger zone is defined by TRIGGER_ZONE_MARGIN_RATIO = 0.10 (10% margin on all sides). DOM polling via _tryConnectDom() retries every 500ms up to 60 times (DOM_RETRY_INTERVAL_MS / DOM_MAX_RETRIES). Feature is mobile-only (checked via _isMobileDevice(): maxTouchPoints > 0 or mobile UA regex) and follows the master switch (isEnabled) via chrome.storage.onChanged listener. Exposed as window.DSStudio.MobileSidebarSwipe with lifecycle methods start(), enable(), disable(), destroy().
- **Unit tests** (`test/unit/mobile-sidebar-swipe.spec.js`, new): 16 tests covering module constants, desktop guard, mobile detection (touch + UA), valid swipe, horizontal zone rejection, wrong direction, insufficient distance, vertical-dominant rejection, timeout rejection, missing-button graceful handling, DOM polling, storage listener enable/disable, and vertical zone rejection (top/bottom margins). All 16 pass.
- **Manifest**: Registered `content/mobile-sidebar-swipe.js` in content_scripts.

- **Manifest**: Version 3.1.4 → 3.2.0.

## [3.1.2] - 2026-06-08

### Fixed

- **GoToTop button not appearing after switching conversations (SPA route change)** (`content/go-top.js`): When navigating between conversations in DeepSeek's single-page application, the gotop button silently failed to appear — even though initial page loads worked correctly. Root cause: `_onRouteChange()` triggered by the `MutationObserver` + `popstate` listener used a one-shot 100ms-delayed call to `_injectButton()` (plus `_reattachScrollListener()` and `_evaluateVisibility()`), without any gating or retry. When `.aaff8b8f` (the input-area wrapper) or `._0706cde` (the native go-bottom button) hadn't mounted yet, `_injectButton()` no-oped and returned `false`, and nothing ever tried again. Fixed by routing the route-change callback through `_tryConnectDom()`, which gates on input-area readiness and retries every 500ms up to 120 times (≈60s). The success branch of `_tryConnectDom()` fully covers the old trio (injection → scroll-container reattachment → listener restart → visibility evaluation), so nothing is lost.
- **Version 3.1.2 retains `[GoToTop-DEBUG]` logs for user verification** — they were removed in 3.1.3 after confirmation.

### Removed

- **Dead code** (`content/go-top.js`): `_reattachScrollListener()` was confirmed to have zero callers (its only former caller was already replaced by the `_tryConnectDom()` success branch in the 3.1.0 fix). The method and its JSDoc comment block were deleted.

### Changed

- **Unit tests** (`test/unit/go-top.spec.js`): Replaced 1 stale test (that asserted on the old `setTimeout`-based one-shot injection trio) with 6 new tests verifying that `_onRouteChange()` drives gated `_tryConnectDom()` retry on route change — wrapper absent → retry scheduled, wrapper appears → injection succeeds, wrapper present → immediate injection, native button present → immediate injection, and old button cleaned up before re-injection. Suite: 893 tests across 47 files, all passing.
- **Manifest**: Version 3.1.1 → 3.1.2.

## [3.1.0] - 2026-06-08

### Fixed

- **Go-to-top button rendering in the wrong place when opening an existing conversation** (`content/go-top.js`): Opening a conversation directly by URL frequently rendered the gotop button as a floating overlay in the wrong position, while a page refresh rendered it correctly — a race condition. Root cause: `_tryConnectDom()` injected the button as soon as `_getAnchor()` (the first message) existed, but the bottom input-area wrapper `.aaff8b8f` and the native go-bottom button `._0706cde` are rendered slightly later. With both anchors still missing, `_injectButton()` fell through to the fixed fallback (`_injectAsFallback()` / `dsw-gotop--fixed`), which appended the button to the first `.ds-theme` element — a `ds-notification-container ds-theme` notification overlay — and nothing ever recovered it. The fix re-gates injection so the button is created ONLY once `.aaff8b8f` or the native button is present; the retry cap was raised from 20 to 120 (`RETRY_INTERVAL` unchanged at 500 ms, ≈60 s total), and on give-up NO button is injected at all (the old "inject a degraded button after the cap" branch was removed). The entire fixed-fallback path — `_injectAsFallback()`, the `dsw-gotop--fixed` CSS rule, and every `_injectionMode === 'fixed'` branch — was deleted; when neither anchor exists, `_injectButton()` now no-ops and returns false.

### Changed

- **Go-to-top scroll is now a toggle and stays clickable** (`content/go-top.js`): Previously the button was disabled (`aria-disabled="true"`) for the duration of the scroll-to-top animation. It now stays enabled (`aria-disabled="false"`) throughout, and clicking it again while a scroll is in progress STOPS the scroll at its current position (settling the active scroll with reason `'stopped-by-user'`) instead of being ignored. A subsequent click starts a fresh scroll. Route-change abort (reason `'aborted'`) is unchanged.
- **Unit tests** (`test/unit/go-top.spec.js`): Removed the obsolete `_injectAsFallback` suite and the fixed-fallback / during-scroll-disabled assertions; added coverage for the new injection gating (no injection until `.aaff8b8f` or native button present, 120-retry give-up with no button, mid-retry `disable()` cancellation, retry-count reset on success), the no-fallback guarantee in `_injectButton`, the never-add-`dsw-gotop--fixed` invariant, and the scroll toggle (aria-disabled stays false; second click aborts with `'stopped-by-user'` and does not restart). Suite: 888 tests across 47 files, all passing.

## [3.0.0] - 2026-06-07

### Added

- **Standalone prompt editor window** (`popup/editor/editor.html|css|js`, new): Prompt content editing moved out of the popup into an independent 1280×720 window opened via `chrome.windows.create({ type: 'popup' })`. Query-string contract: `?target=global` edits the global default prompt; `?target=preset&id=<presetId>` edits that preset's content. The window title shows 「全域預設提示詞」 or the preset name, with the familiar 「已儲存」 save-status indicator. Auto-save is defensive against OS-level window closing: dirty flag + debounced input save (600 ms) + `blur` + `visibilitychange` + `pagehide`, all routed through `StorageManager` (`saveGlobalDefaultPrompt` / `saveOnePromptPreset`). Preset saves additionally broadcast `ACTIVE_PRESET_CHANGED` so an open DeepSeek tab refreshes immediately. Invalid targets and deleted presets render a disabled state. No new permissions required.
- **Global prompt injection toggle** (`#globalPromptToggle`): A switch identical in appearance to the master toggle, right-aligned on the Global Prompt card (vertically lined up with the master toggle above). New storage key `globalPromptEnabled` (sync, default `true` — existing users keep current behavior). Gating in `content-script.js` `buildInjectionPrefix()`: the global default prompt is injected only when BOTH the master switch and this toggle are on; the master switch retains highest priority via the existing `injectPrefix()` early return. Included in JSON backup export/import (`getSettings()` / `restoreSettings()`).
- **Shared messaging utility** (`utils/messaging.js`, new): `broadcastActivePreset(presetId, presetContent)` extracted from popup.js's tab-broadcast logic (active-tab query, `chat.deepseek.com` filter, silent error swallowing), exposed as `window.DSVMessaging` + guarded `module.exports`, reused by both popup.js and editor.js.

### Changed

- **Popup layout refactor** (`popup/popup.html|css|js`): Both prompt-editing textareas (`#globalDefaultPrompt`, `#promptInput`) removed. The Global Prompt card is now a single row — label + pencil button (`#editGlobalPromptBtn`, `.icon-btn` with an inline SVG pencil drawn upper-right → lower-left) on the left, the new toggle on the right. The Prompt Group controls are reordered to [shortened dropdown (`max-width: 150px`)] [+ `#addPresetBtn`] [pencil `#editPresetBtn`]; the preset pencil is disabled when no preset is selected (replacing the old disabled-textarea behavior) and opens the editor window for the active preset. The Backup & Restore card moved from the right column to the left column below Prompt Group. Editor windows are singletons per target — a second click focuses the existing window (`chrome.windows.update`), recreating it only if it was closed. The now-obsolete `initColumnLayout()`, content dirty flags, and `saveCurrentPresetContent()` were removed; `sendActivePresetToContentScript()` delegates to `DSVMessaging`.
- **Unit tests**: `popup.spec.js` rewritten (column-layout suite replaced by pencil disabled-state + editor-URL tests); new specs `content-script.global-prompt-gating.spec.js` (9), `storage-manager.global-prompt-enabled.spec.js` (6), `messaging.spec.js` (7), `editor.spec.js` (~25), `popup-editor-window.spec.js` (10); `storage-manager.migration-push.spec.js` helper updated for the new DEFAULTS key. Suite: 887 tests across 47 files, all passing.

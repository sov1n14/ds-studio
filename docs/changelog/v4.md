## [4.7.2] - 2026-07-11

### Fixed

- **`_get()` pin-on-read override removed** (report.md §4.2 Step 2): Previously, if a key was parked in `dsLocalAuth` (its last sync write failed), `_get()` unconditionally or conditionally overrode the already-computed per-item recency result (`merged[key]`) with the local value — even when the local value was demonstrably older than the cloud copy. This let a stale local edit that once failed to sync permanently shadow a genuinely newer cloud edit made on another device, since parking never expires on its own. The override block (previously `utils/storage-manager.js` lines 269-295) has been removed; `_get()` now returns the pure per-item `updatedAt` recency result computed earlier in the function, with no parking-based layer on top.

### Removed

- **`_shouldPinLocalPreset`** (`utils/storage-manager.presets.js`): dead code after the pin-on-read block was removed — confirmed zero remaining callers.

### Changed

- **`dsLocalAuth` semantics**: now exclusively a pending-retry queue for writes that failed to reach `chrome.storage.sync` (`_set()`'s failure branch parks keys, `retrySync()` drains them). It no longer has any influence on what `_get()`/`getSettings()`/`syncNow()` return.

### Tests

- `test/unit/storage-manager.pin-recency.spec.js`: removed the `_shouldPinLocalPreset` pure-helper describe block (6 tests, function deleted); rewrote the `_get()` parked-preset describe block to assert recency is independent of parking.
- `test/unit/storage-manager.sync-conflict.spec.js`: rewrote the `dsLocalAuth` Plan A tracking tests — `_get()` no longer prefers local based on parking alone; added coverage that non-preset keys also ignore parking status.
- Added regression test: a locally-parked failed edit never resurfaces over a later, different, newer cloud edit — verified across three consecutive reads.
- Full `storage-manager*.spec.js` sweep: 203/203 passing, 0 regressions.

## [4.6.2] - 2026-06-28

### Fixed

- **Cloud sync — preset order not propagating across devices**: Added `dsPresetOrderMeta` key (`{ order: string[], orderUpdatedAt: number }`) as the authoritative recency signal for preset ordering. Reordering now bumps `orderUpdatedAt`; `_get()` uses `_pickPresetOrderByRecency()` to select the side with the larger timestamp, so the newest order always wins on read.
- **Cloud sync — preset content silently discarded**: `_get()` now refines `dsLocalAuth` pinning by type — `PRESET_INDEX` is only pinned locally when local `orderUpdatedAt >= sync orderUpdatedAt`; `dsPreset_*` keys are only pinned when local `updatedAt >= sync updatedAt`. Previously, all `dsLocalAuth` keys were blindly preferred over sync.
- **First-sync conflict detection locking second device in read-only mode**: `initialize()` conflict detection is now classified into `none` / `auto` / `manual`. One-directional or clearly timestamped divergence is auto-merged silently (`resolveSyncConflict()`); the conflict Modal is shown only when the same preset ID was modified on both devices with equal `updatedAt` (true ambiguous conflict).
- **Manual sync pushing stale local data over newer cloud data**: `retrySync()` pre-reads cloud snapshots and compares timestamps before pushing each key — it skips pushing `PRESET_INDEX` when cloud `orderUpdatedAt` is newer, and skips `dsPreset_*` when cloud `updatedAt` is newer. After pushing, it calls `resolveSyncConflict()` if `_detectSyncConflict` returns `auto`, so newer cloud changes are pulled in the same operation.
- **`mergePresets` discarding newer content on timestamp tie**: Added `createdAt` as a stable tiebreak — when two versions of a preset have equal `updatedAt`, the one with the earlier `createdAt` wins.

### Added

- **`utils/storage-manager.chatmap.js`** (~400 lines): Extracted all `chatPresetMap` chunk operations (`mutateChatPresetMap`, `saveChatPresetMap`, `getChatPresetMap`, `bindChatToPreset`, `unbindChat`) from `storage-manager.presets.js` into a new dedicated mixin bundle `__DS_StorageManager_chatmap`. Reduces `storage-manager.presets.js` from 537 to 117 lines.
- **`StorageManager._detectSyncConflict(syncRaw, localRaw)`** (`storage-manager.sync.js`): New synchronous method returning `'none'` | `'auto'` | `'manual'`. Used by `initialize()` and `retrySync()`.
- **`StorageManager._pickPresetOrderByRecency(localMeta, syncMeta)`** (`storage-manager.presets.js`): New helper returning the meta with larger `orderUpdatedAt`, or `null` on a tie.
- **`CHANGELOG.md`** (project root): Bootstrapped Keep-a-Changelog file for this project.

### Changed

- **`mergePresets`** upgraded to 4 parameters: `(basePresets, newPresets, baseOrderMeta, incOrderMeta)`. The `baseOrderMeta`/`incOrderMeta` params are optional — 2-param calls remain fully backward compatible (insertion order is preserved).
- **`savePromptPresets(presets, orderMeta?)`**: Accepts optional `orderMeta`; when the ID order changes, writes `dsPresetOrderMeta` alongside `dsPresetIndex`.
- **`resolveSyncConflict()`**: Now reads both sides' `PRESET_ORDER_META`, passes 4-param metas to `mergePresets`, computes a merged meta with `Math.max` timestamp, and deletes `PRESET_ORDER_META` from the raw updates spread to prevent overwriting the freshly saved value.
- **`getSettings()`**: Excludes `dsPresetOrderMeta` from both the fetch key list and the returned settings object (internal implementation detail, not a user-facing setting).
- **`popup/popup.js` `onReorder`**: Now passes explicit `{ order, orderUpdatedAt: Date.now() }` as the second argument to `savePromptPresets`.

### Tests

- Updated `storage-manager.resolve-sync-conflict.spec.js`: Order meta propagation, `PRESET_ORDER_META` not overwritten by raw updates spread.

## [4.5.1] - 2026-06-18

### Fixed

- **Temporary Conversation — deletion scope (`to-do/to-fix/report.md`)**: The toggle no longer deletes *any* conversation the user leaves while enabled. A conversation is now treated as "temporary" (deletable) ONLY if it was newly created while the toggle was ON — detected authoritatively by an HTTP request to `/api/v0/chat_session/create`. Opening a historical conversation (which calls `/api/v0/chat/history_messages`) is never marked and never deleted. The tracked temporary conversation UUID is persisted in `sessionStorage` (`dss-temporary-chat-uuid`) so it survives a refresh.
- **Temporary Conversation — address-bar navigation (Bug)**: Pressing Enter in the address bar on the *current* URL (or any navigation whose destination equals the current URL, plus reloads) no longer deletes the conversation. Deletion now requires leaving to a genuinely different URL. This is determined by comparing the Navigation API destination URL against the current URL and checking `navigationType === 'reload'`, with a suppression flag that also blocks the subsequent `beforeunload` delete.
- **Temporary Conversation — toggle scope on non-homepage URLs (Supplementary requirement)**: The toggle row is now removed from the DOM whenever the SPA navigates away from the homepage (`pathname !== '/'`) and re-injected on return. Removal does not alter the persisted on/off state.
- **Temporary Conversation — toggle-off still deletes an already-tracked conversation**: Turning the toggle OFF stops marking new conversations but keeps the navigation / `beforeunload` listeners attached as long as a tracked temporary conversation exists, so leaving it still deletes it. Listeners detach only once nothing is tracked and the toggle is OFF.

### Added (diagnostics)

- **`[DV:TempChatToggle]` logging**: Per `report.md`, the homepage toggle now emits diagnostic console logs at init / navigation / anchor-lookup / inject / remove / observer-disconnect to help investigate intermittent "toggle missing until refresh" reports in a real browser.

### Changed (code)

- **`content/censor-xhr-hook.js`** (99 → 123 lines): In addition to capturing the `Authorization` header, it now detects requests to `/api/v0/chat_session/create` (via both `XMLHttpRequest.prototype.open` and a `window.fetch` wrapper) and broadcasts `window.postMessage({ type: 'DSS_CHAT_CREATE_DETECTED' })`.
- **`content/temporary-chat-constants.js`** (23 → 31 lines): Added `DSS_TEMP_CHAT_UUID_KEY = 'dss-temporary-chat-uuid'`, `DSS_CHAT_CREATE_MESSAGE_TYPE = 'DSS_CHAT_CREATE_DETECTED'`, and `DSS_CHAT_CREATE_ENDPOINT = '/api/v0/chat_session/create'`. Removed the now-dead `DSS_CHAT_LEFT_EVENT`.
- **`content/temporary-chat-delete.js`** (249 → 373 lines): Reworked to mark only created-while-enabled conversations, track the UUID in `sessionStorage`, delete only on a genuine leave (different URL) or tab close, suppress same-URL and reload deletes, and keep deleting an already-tracked conversation after the toggle is turned off.

## [4.2.2] - 2026-06-14

### Added

- **Mobile overlay position race-condition fix** (`to-do/to-fix/report.md`): On mobile viewport (< 768 px), page reload could leave the preset-dropdown overlay stuck at an incorrect left position because `reposition()` ran before the DeepSeek new-chat/share buttons (`._1aa2651`) had finished layout.

  **Root cause**: The existing `ResizeObserver` (container only), `window.resize` listener, and `MutationObserver` all fail to detect when the new-chat button's `left` shifts from ~160 px to ~189 px as sibling elements settle during page load — the button's border-box width (84 px) never changes, only its x-position relative to the viewport.

  **Fix**: Added a bounded settle retry loop (`preset-settle.scheduler.js`) that polls `buttonRect.left` every animation frame and re-applies `reposition()` until the metric stabilizes for K consecutive frames (converged) or hits a frame budget (maxFrames). Triggered once at mount time (`startSettle('initial-settle')`) in `preset-overlay.controller.js`.

  **Defense-in-depth**: Existing container RO + window-resize listener are retained (they cover real resizes and 768 px boundary crossing). No new RO on the button is added (RO only fires on border-box size changes — the button's width is constant).

- **New module**: `content/preset-settle.scheduler.js` — pure timing/control-logic module (no DOM access) that manages a settle loop: `apply` → `measure` → compare → stop on converge / maxFrames / cancel / detached.

### Changed

- **`content/preset-overlay.controller.js`**: Added `_settle` state field, `startSettle(reason)` method, mount-time trigger (`mountTo()` calls `startSettle('initial-settle')` at the end), and unmount teardown (`unmount()` cancels and nulls `_settle`).

- **`manifest.json`**: Registered `content/preset-settle.scheduler.js` in `content_scripts[0].js` (after resolvers, before controller). Version bumped from `4.2.1` to `4.2.2`.

- **`test/setup/vitest.setup.js`**: Added side-effect import of `preset-settle.scheduler.js` in the overlay dependency chain.

### Added (tests)

- **`test/unit/preset-settle.scheduler.spec.js`** (14 tests): Validates guard clauses (8), delayed right-shift convergence, first-frame stable fast path, maxFrames budget, mid-execution cancel, null-measure detached stop, and initial-null wait.

- **`test/unit/preset-overlay.controller.spec.js`**: Extended with 2 new test cases — mountTo triggers settle loop and unmount does not crash during settle.

## [4.0.0] - 2026-06-13

### Changed

- **Large-file modularization refactor** (`to-do/to-implement/requirement.md`): Five oversized source files and one stylesheet were split into smaller, single-concern modules to comply with the `coding-guidelines` file-size thresholds (JS 450 proactive / 600 absolute; CSS 350 / 500). Runtime behavior and every public API are **unchanged** — the split is purely structural. All 985 unit tests across 49 files remain green; no test assertions were modified.

- **Dual-load split pattern**: Each oversized singleton keeps its original filename as a state-bearing **entry file**; cohesive method groups move into **bundle files**. Bundle files attach their method group to a `globalThis.__DS_*` key (guarded by `if (typeof module !== 'undefined' && module.exports)` for Vitest); the entry file runs `Object.assign(Singleton, globalThis.__DS_* )` to merge them before attaching to `window` / `module.exports`. `content-script` and `popup` helpers that close over mutable state use a `createX(ctx)` factory with live getter/setter callbacks so cross-file state stays in sync. No ES `import`/`export` is introduced; classic-script (production) and Vitest (test) load paths are both preserved.

- **`utils/storage-manager.js` split** (1199 → 546 lines entry):
  - `storage-manager.js` (546) — entry: `KEYS`/`DEFAULTS`/`errors`, safe wrappers, `getSettings`, `initialize`, shared state
  - `storage-manager.chunking.js` (77) — ChatPresetMap physical chunking & reconciliation
  - `storage-manager.lock.js` (132) — cross-context advisory lock (Method C)

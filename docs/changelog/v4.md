## [4.9.0] - 2026-07-11

### Changed

- **Two-layer temporary-conversation deletion architecture**: Real-time deletion and crash/shutdown remediation are now fully separated.
  - **Layer 1 (real-time, content script)**: `beforeunload` (tab/browser close) now calls `fetch(..., { keepalive: true })` directly instead of relaying through the Service Worker via `chrome.runtime.sendMessage` — removing the unreliable IPC hop that could fail to arrive during tab teardown. An entry is removed from the pending-delete queue only after the delete API confirms success.
  - **Layer 2 (remediation, Service Worker, `onStartup`-only)**: The existing `chrome.runtime.onStartup` listener (previously used only for cloud-preset sync) is extended with pending-delete remediation — on every browser restart, it reads the shared queue and retries each entry with this device's own locally-cached auth token, removing an entry only on confirmed success and incrementing `attemptCount` otherwise.
- **Cross-device pending-delete queue as single source of truth**: The pending-delete queue moved to `chrome.storage.sync` (`dss-pending-deletes-sync`, `{ chatUuid, attemptCount }`) with no separate local-only copy — any device signed into the same Chrome account can remediate any queue entry using its own locally-captured auth token, whether or not it originally tracked that conversation.
- **Sync-Change Safeguard**: A `chrome.storage.onChanged` (sync area) listener triggers an additional remediation sweep to mitigate the `onStartup` cold-start hydration race, guarded by a new device-local open-set (`dss-open-temp-uuids`, local-only) so the currently-open conversation on this device is never deleted by the sweep.

### Privacy

- `authToken` is cached only in `chrome.storage.local` (`dss-last-auth-token`) and never syncs to any other device. Only the non-sensitive `chatUuid` and `attemptCount` are written to `chrome.storage.sync`.

### Docs

- `PRIVACY.md` / `docs/PRIVACY.md`: documented the cross-device pending-delete queue's data model and the local-only auth-token guarantee.
- `docs/FEATURES.md` / `docs/spec/04-features.md`: documented the privacy-guarantee scope (guaranteed vs. delayed-until-next-launch scenarios) and the known-limitations table for temporary-conversation deletion.

### Version

- **`manifest.json`**: Bumped `4.8.5` → `4.9.0` (minor — new cross-device remediation architecture).

## [4.8.3] - 2026-07-11

### Fixed

- **Deleted prompt groups resurrected across devices after popup reopen**: deletion previously only removed a preset's own key from the *deleting device's own* sync/local storage — it left no marker other devices could recognize. Other devices' stale local `dsPresetIndex` still listed the deleted id, and on next popup open, `resolveSyncConflict()` → `mergePresets()` treated that stale id as a legitimate item to keep (pure id union, no concept of intentional deletion), resurrecting it and re-pushing it to the cloud.
- **New tombstone mechanism** (`utils/storage-manager.tombstones.js`, new bundle): `_mergeTombstones`, `_pruneTombstones` (30-day retention), `_isTombstonedAway` (pure decision fn), `recordPresetTombstones()`. New storage key `dsPresetTombstones: { [presetId]: deletedAt }`, persisted to both `local` and `sync` via the existing guarded `_set()` (reusing the v4.8.2 8KB size-guard/retry-queue gate unmodified).
- `savePromptPresets()` (`utils/storage-manager.presets.js`) now calls `recordPresetTombstones(deletedIds)` immediately after removing deleted preset keys.
- `mergePresets()` (`utils/storage-manager.presets.js`) gained an optional `tombstones` parameter: an id is dropped from either side of the merge when its tombstone's `deletedAt` is not older than that side's `updatedAt` — a stale untouched copy is excluded, but a later edit with a newer `updatedAt` than the tombstone correctly survives (no over-deletion).
- `resolveSyncConflict()` (`utils/storage-manager.sync.js`) now reads local+sync `dsPresetTombstones`, merges and prunes them, passes the merged map into `mergePresets()`, and persists the merged tombstones back to both storage areas.
- **`_get()` index-persistence gap closed** (`utils/storage-manager.js`): when the sync-side `dsPresetIndex`/`dsPresetOrderMeta` wins by recency, it is now added to `remoteWinsToPersist` (same mechanism already used for `dsPreset_*` items) so the winning index is actually written to `chrome.storage.local` instead of staying in-memory-only for the remainder of the popup session — this closed the second half of the root cause (a stale on-disk snapshot no longer survives to the next popup open).
- `manifest.json`, `popup/popup.html`, `popup/editor/editor.html`: added script/content-script entry for `storage-manager.tombstones.js`.

## [4.8.2] - 2026-07-11

### Fixed

- **8KB payload-size guard in front of the `dsLocalAuth` retry queue** (report.md §4.2/§6 Step 6, final step of the sync-refactor plan): `_set(items)` in `utils/storage-manager.js` now splits every write batch per-key by serialized UTF-8 byte size before attempting `chrome.storage.sync.set()`. Keys whose `{ [key]: value }` payload exceeds `QUOTA_BYTES_PER_ITEM` (8192 bytes) are diverted before the sync call ever fires: written to `chrome.storage.local` only (value never lost), tracked in a new local-only `dsOversizedKeys` list, and explicitly excluded from `dsLocalAuth`. Previously every write failure — transient (`context-invalidated`, offline, rate-limit) or permanent (oversized) — was treated identically and parked in `dsLocalAuth` for indefinite retry by `retrySync()`; an oversized item could never actually succeed, silently wasting the sync write quota and diverging across devices forever while `retrySync()` kept "trying."
- **`_byteLen()` UTF-8 accuracy**: changed from `JSON.stringify(obj).length` (JS string length — undercounts multi-byte characters) to `new TextEncoder().encode(JSON.stringify(obj)).length` (actual UTF-8 byte count). Chrome's `QUOTA_BYTES_PER_ITEM` is byte-based, so Chinese-heavy prompt content could previously slip past a size check using the naive character count.
- **Self-healing tracking**: a key already in `dsOversizedKeys` is automatically removed the next time it's written at a size at or under the 8192-byte limit.
- **`hasOversizedItems()`** (new, `utils/storage-manager.sync.js`): returns `true` iff `dsOversizedKeys` is non-empty.
- **`popup.js`'s `refreshSyncStatus()`**: now also checks `hasOversizedItems()` and shows a distinct status text (`dsI18n.t('syncStatusOversized')`, new i18n key in `utils/i18n.js`: zh_TW `內容過大，僅存本機`, en `Too Large — Local Only`), taking precedence over the normal synced/unsynced text so a permanent failure is never confused with a transient pending state.
- `retrySync()` required no changes — oversized keys are filtered out of `dsLocalAuth` unconditionally inside `_set()`, so they can never enter the retry queue in the first place.

### Tests

- New `test/unit/storage-manager.oversize-guard.spec.js` (18 tests): per-key oversize splitting (mixed batch and all-oversized batch), oversized keys never entering `dsLocalAuth` (including stripping a pre-existing leftover), value preservation + `dsOversizedKeys` tracking, self-healing, `hasOversizedItems()`, and `_byteLen()`'s UTF-8 byte accuracy versus the naive-length undercount bug this change fixes.
- Extended `test/unit/popup.sync-write-quota.spec.js` (+4 tests): `refreshSyncStatus()`'s oversized-status precedence over synced/unsynced.
- Full regression re-run across all sync/quota-adjacent specs: 96/96 pass, no regressions.

## [4.7.4] - 2026-07-11

### Fixed

- **File-split closure regression in `_installChunkCacheInvalidator()`**: When `initialize()`/`_installChunkCacheInvalidator()` were extracted into the new `utils/storage-manager.init.js` (v4.7.3 split), the invalidator's `onChanged` listener kept a bare `StorageManager.xxx` reference instead of using `this`.
